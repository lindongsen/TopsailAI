# Golang Development rules

## Project Structure

It should include the following:

- folder: `docs`, store documention files
- folder: `cli`, store script files
- folder: `src`, store source code files, folder `src/main/` store package `main`
- folder: `tests`, store test files for (functional-test, integration-test)
- file: `README.md`, save detail info about of the project

## Log format

Format the output log content using JSON and use the standard library `log/slog` + `lumberjack`(for logrotate).

A standardized JSON log entry should include the following core fields to ensure consistency and searchability:

- timestamp: The timestamp generated by the log.
    Format: The ISO 8601 standard format must be used and standardized to UTC time, such as 2023-10-05T14:30Z.
    Function: Facilitating precise queries and sorting by time range.

- level: The severity level of the log.
    Common values: DEBUG, INFO, Warning, ERROR, FATAL.
    Function: Used for quick filtering, such as focusing only on logs at the level of Warning and above in production environments.

- message: The main content of the log, used to describe the details of the event.
    Function: Provide human readable event descriptions.

- module: Name of the module/function that generates logs

- trace_id: Distributed Trace ID.
    Function: This is a key field in the cloud native environment. By passing the same trace.id throughout the entire request chain, logs scattered across different services can be correlated, fully restoring the processing of a request and being the core of quickly locating cross service issues.

Required:
    When the program starts, parameters can be specified to choice STDOUT or FILE to output logs.
    When specifying the output of logs to a file, `logrotate` must be set or set by default.
