# Issue: Audit log records for create operations may fail to persist

## Problem

`services.AuditLogService.Log` requires `ResourceID` to be non-empty:

```go
if req.ResourceID == "" {
    return nil, fmt.Errorf("resource_id is required")
}
```

However, the audit middleware in `internal/api/middleware/audit.go` passes an empty `ResourceID` for POST create operations (e.g., `POST /api/v1/accounts`, `POST /api/v1/groups`), because the resource ID is generated by the handler/service and is not yet known when the middleware runs.

The middleware silently ignores the resulting error in the async goroutine:

```go
go func() {
    _, _ = auditSvc.Log(...)
}()
```

As a result, real audit log records for create operations may fail to persist without any visible error.

## Affected Operations

- `POST /api/v1/accounts`
- `POST /api/v1/groups`
- `POST /api/v1/groups/:group_id/members`
- `POST /api/v1/groups/:group_id/messages`
- `POST /api/v1/accounts/:account_id/api-keys`
- Any other POST endpoint that creates a resource whose ID is generated server-side

## Proposed Options

### Option A: Relax validation for create actions
Allow `ResourceID` to be empty when `action == "create"`, and store an empty or placeholder value. This is the smallest change but loses the resource association for create events.

### Option B: Have handlers set the created resource ID back into the context
After a successful create, handlers store the new resource ID (and optionally name) into `gin.Context` so the audit middleware can read it. This keeps audit records complete but requires touching every create handler.

### Option C: Generate a placeholder ID
Use a deterministic placeholder (e.g., `"pending"` or a generated UUID) for create actions and update it later. More complex and may require schema/supporting changes.

## Recommendation

Option B is preferred because it preserves full audit traceability and keeps the validation rule intact. A helper such as `middleware.SetAuditResource(ctx, id, name)` could be added and called by create handlers after the resource is created.

## Related Files

- `internal/api/middleware/audit.go`
- `internal/services/audit_log_service.go`
- All create handlers under `internal/api/handlers/`

## Notes

This issue was discovered during the review of `issue-unit-test-audit-middleware.md`. The unit tests use a mock audit service, so this inconsistency is not caught by the current test suite.
