# FAQ

## What is the difference between `DeleteAtMs` and `DeletedAt` in the `group_messages` table?

| Field | Database Column | Type | Purpose |
|-------|-----------------|------|---------|
| `DeleteAtMs` | `delete_at_ms` | `int64` (milliseconds timestamp) | **Business-level "message recall" (withdraw message)**. Set when the API `DELETE /api/v1/groups/:group_id/messages/:message_id` is called. The handler clears `message_text` / `message_attachments`, sets `is_deleted = true`, and records the recall timestamp. This matches the "withdraw message" logic described in ORIGIN.md. |
| `DeletedAt` | `deleted_at` | `gorm.DeletedAt` | **GORM built-in soft delete**. If this field is populated, GORM treats the row as logically deleted and hides it from normal queries. |

### Differences in Actual Usage

1. **`DeleteAtMs` is actively used**: It is used for the message recall (withdraw) feature, preserving the message record while clearing its content.
2. **`DeletedAt` is not used in message processing**: Although this column exists in the database schema (generated by GORM auto-migration), the message lifecycle management does not depend on it. Currently, it is only checked in the deletion logic for `Group` and `GroupMember` entities.

### Discrepancy with ORIGIN.md

ORIGIN.md only documents:
- `is_deleted`
- `delete_at_ms`

It does **not** mention `deleted_at`. The `deleted_at` column is an implicit artifact generated by GORM and currently serves no active business purpose for the `group_messages` table.
